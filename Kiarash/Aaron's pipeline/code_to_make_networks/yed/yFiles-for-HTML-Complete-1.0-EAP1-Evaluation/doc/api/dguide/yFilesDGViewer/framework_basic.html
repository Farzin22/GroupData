Basic Features

Modules

A module corresponds roughly to a namespace in languages like Java and C#. It is emulated in Javascript by using nested objects.

The yFiles Class Framework provides the function yfiles.module to encapsulate the logic of creating a new namespace object, or appending to an existing one. This function creates the specified namespace if it does not yet exist, and invokes the namespace object's callback function. The callback function can then attach functions, types, properties or fields to that object and they will be available in the specified object.

Example 3.1. Creating and using modules

yfiles.module('my.module', function(exports) {
            exports.doSomething = function() { return 'Hello World'; };
          });
          my.module.doSomething() === 'Hello World'; // true

While not strictly required, using modules helps to reduce boilerplate code and prevent pollution of the global namespace.

Native Extensions

The yFiles Class Framework tries very hard not to add too many methods to native objects. Still, it is worth noting that since the yFiles Class Framework requires an environment that supports Ecmascript 5 or greater, most of the major objections against extending native objects are no longer valid.

The following two methods were too basic not to include at the Object.prototype level: equals and hashCode. Should a future Ecmascript standard define these methods, it is very likely that they will have the same semantics as the ones we have defined.

equals
Tests whether this object is equal to another object. The default implementation first checks that both objects are of the same type, then uses reference equality, thus ensuring that non-identical objects are never equal. When implementing a specialized version of this method for a subtype, it is important to also implement a specialized hashcode method that creates the hashcode based on the same fields that are used for checking equality.
hashCode
Provides a hash code for the object. A hash code is an integer that should be mostly unique and well distributed over a given class type. It is often used as a dictionary key.

The yFiles Class Framework specifies both of these methods as "not-enumerable", so that they will not affect naive for(... in ...) loops.

There are specialized variants for String, Number, and Boolean.
